Abstract Factory

(Fábrica abstracta)

Propósito

Abstract Factory (Fábrica abstracta en español) tiene como objetivo proporcionar una interfaz para crear familias de objetos relacionados o dependientes sin especificar sus clases concretas. Este patrón es útil cuando un sistema debe ser independiente de cómo se crean, componen y representan sus productos.

Problema

Imagina que estás desarrollando una aplicación de interfaz de usuario (UI) que debe ser compatible con múltiples sistemas operativos, como Windows, macOS y Linux. Cada sistema operativo tiene su propio conjunto de componentes de interfaz, como botones, menús y cuadros de diálogo, que deben mantener un estilo consistente para proporcionar una experiencia de usuario coherente.

En tu aplicación, quieres asegurarte de que si el sistema operativo es Windows, se usen botones, menús y cuadros de diálogo con estilo Windows. Si es macOS, se deben usar los correspondientes componentes con estilo macOS, y lo mismo para Linux. Sin embargo, no quieres que el código de tu aplicación esté lleno de condiciones para verificar el sistema operativo y crear los componentes apropiados.

Solución

Para resolver el problema de mantener una interfaz de usuario consistente y adaptable en diferentes sistemas operativos, el patrón Abstract Factory proporciona una solución elegante al definir una interfaz para crear familias de objetos relacionados o dependientes sin especificar sus clases concretas. Implementando fábricas concretas para cada sistema operativo (como Windows, macOS y Linux), y definiendo interfaces abstractas para los componentes de la UI (como Botón, Menú y Cuadro de diálogo), podemos crear componentes específicos del sistema operativo de manera consistente. El código cliente interactúa con la interfaz de la fábrica abstracta, lo que permite la creación de componentes sin acoplarse a sus implementaciones concretas, facilitando así la extensibilidad y el mantenimiento del software.

Pros y contras

Pros

  • Garantiza que los productos creados por las fábricas sean compatibles entre sí. Por ejemplo, todos los componentes de la interfaz de usuario en un sistema operativo tendrán un estilo uniforme.
  • El cliente utiliza interfaces y clases abstractas, lo que disminuye la dependencia de clases concretas. Esto mejora la flexibilidad y facilita cambios y extensiones en el futuro.
  • Nuevos tipos de productos o familias de productos se pueden añadir fácilmente mediante la creación de nuevas fábricas concretas sin afectar al código cliente.

Contras

  • Introducir el patrón Abstract Factory puede incrementar la complejidad del código debido a la necesidad de crear múltiples interfaces y clases concretas.
  • La implementación de este patrón puede resultar en la creación de muchas clases, lo que puede hacer que el sistema sea más difícil de manejar y mantener.
  • Para desarrolladores novatos o en proyectos más pequeños, el uso de este patrón puede parecer innecesariamente complicado y ser difícil de comprender.
