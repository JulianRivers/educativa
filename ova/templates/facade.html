Facade

Fachada

Propósito

Facade El patrón de diseño Facade (o Fachada) es un patrón estructural que proporciona una interfaz simplificada para un conjunto complejo de interfaces en un sistema. Este patrón oculta la complejidad de las subsistemas y expone solo una interfaz fácil de usar.

Su propósito es proporcionar una interfaz más simple y clara para interactuar con sistemas complejos. Facilita a los clientes el uso de subsistemas complejos y desacoplar el código cliente del subsistema subyacente. Permite cambiar el subsistema sin afectar el código cliente.

Problema

Imagina que estás desarrollando un sistema de gestión para un hogar inteligente. Este sistema permite a los usuarios controlar varios dispositivos como luces, termostatos, cámaras de seguridad y sistemas de entretenimiento. Cada dispositivo tiene su propia interfaz compleja con múltiples funciones. Sin embargo, los usuarios quieren una forma sencilla de controlar todos estos dispositivos sin tener que interactuar directamente con cada uno.

Solución

Para resolver el problema de gestión de un hogar inteligente utilizando el patrón de diseño Facade, implementamos una clase SmartHomeFacade que proporciona una interfaz simplificada para controlar varios dispositivos inteligentes, tales como LightManager, ThermostatManager, SecurityCameraManager, y EntertainmentSystemManager. La clase SmartHomeFacade encapsula las complejidades de cada dispositivo y expone métodos fáciles de usar para las operaciones comunes, como encender y apagar todas las luces, ajustar la temperatura del termostato, iniciar la grabación de todas las cámaras de seguridad, y reproducir música en el sistema de entretenimiento. De esta manera, los usuarios pueden interactuar con el sistema de hogar inteligente a través de una única interfaz simplificada, sin necesidad de conocer los detalles internos de cada dispositivo.

Pros y contras

Pros

  • El patrón Facade proporciona una interfaz simplificada y unificada para interactuar con un sistema complejo. Esto facilita la comprensión y el uso del sistema para los clientes.
  • Encapsula las complejidades internas del subsistema, permitiendo a los desarrolladores trabajar con una interfaz más manejable y reduciendo la carga cognitiva.
  • Proporciona un único punto de control para las operaciones del subsistema, lo que facilita la localización de errores y la implementación de cambios.

Contras

  • Si la clase Facade crece demasiado, puede convertirse en un "Dios objeto" con demasiadas responsabilidades, lo que contradice el principio de responsabilidad única y puede dificultar su mantenimiento.
  • Introduce una capa adicional de abstracción, lo que puede añadir complejidad al diseño del sistema. Esto puede ser innecesario en sistemas simples donde el uso de Facade no aporta mucho beneficio.
