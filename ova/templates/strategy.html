Strategy

Estrategia

Propósito

Strategy El patrón de diseño Strategy es un patrón de comportamiento que permite definir una familia de algoritmos, encapsular cada uno de ellos y hacerlos intercambiables. Esto permite que el algoritmo varíe independientemente de los clientes que lo usan. En otras palabras, el patrón Strategy permite seleccionar un algoritmo en tiempo de ejecución.

El propósito principal del patrón Strategy es separar el comportamiento de una clase del cliente, permitiendo que este comportamiento se pueda cambiar sin modificar la clase del cliente. Este patrón es especialmente útil cuando una clase tiene múltiples comportamientos que pueden ser elegidos y cambiados dinámicamente.

Problema

Imaginemos que estamos desarrollando una aplicación que ofrece diferentes métodos de pago (tarjeta de crédito, PayPal, criptomonedas). Queremos que el usuario pueda seleccionar su método de pago preferido en tiempo de ejecución y que cada método de pago tenga su propia lógica de procesamiento.

Solución

Para resolver este problema utilizando el patrón Strategy, definiremos una interfaz PaymentStrategy con un método pay(amount). Luego, crearemos varias estrategias concretas, como CreditCardPayment, PayPalPayment y CryptoPayment, cada una implementando la interfaz PaymentStrategy con su propia lógica de pago. La clase ShoppingCart actuará como el contexto, manteniendo una referencia a una instancia de PaymentStrategy y utilizando esa instancia para realizar el pago.

Pros y contras

Pros

  • Permite cambiar los algoritmos o comportamientos en tiempo de ejecución.
  • Separa el comportamiento del contexto, lo que facilita la adición de nuevas estrategias sin modificar el contexto.
  • Promueve la reutilización de código al encapsular algoritmos en clases separadas.

Contras

  • Puede introducir sobrecarga adicional si se crean demasiadas estrategias pequeñas.
  • Incrementa la complejidad del sistema al añadir múltiples clases y la necesidad de mantener las interfaces y sus implementaciones.
